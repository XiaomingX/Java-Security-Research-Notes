# Mutation XSS 详解

本文用简单易懂的语言介绍 Mutation XSS（变异 XSS）的基本原理、利用思路以及防护时可能遇到的问题，并附上实际的例子和代码示例，帮助大家理解这种利用 HTML 解析器差异绕过过滤的攻击方式。

---

## 1. 什么是 Mutation XSS？

一般的 XSS 漏洞往往是因为没有对用户输入进行正确的 HTML 转义；但有些场景（例如富文本编辑器）需要允许部分格式化的 HTML（比如 `<a>`、`<strong>` 等），这时服务器就必须对用户提交的 HTML 做过滤，只保留“合法”的标签。

**Mutation XSS 就是利用下面这个事实：**

- **过滤器（清理器）解析 HTML 时**：先把输入解析成 DOM，再遍历删除危险标签/属性，然后序列化成 HTML 字符串。
- **浏览器解析 HTML 时**：可能由于上下文不同或者命名空间（HTML、SVG、MathML）等因素，以和过滤器不一样的方式解释同一段代码。

也就是说，过滤器认为某段内容安全，但浏览器解析时却将其变成了可执行的恶意代码，从而触发 XSS。

---

## 2. 基本利用思路

### 2.1 利用解析环境的差异

- **标签上下文不同**  
  某些标签（例如 `<title>` 或 `<noscript>`）有特殊的解析规则。  
  **举例**：在 `<title>` 标签内，浏览器只把内容当成纯文本，不解析内部的 HTML 标签；但如果清理器不知道这是 `<title>` 标签，就可能误把嵌入的“标签”当作普通属性值，导致后续被浏览器重新解释为标签，从而执行恶意代码。

- **命名空间混淆**  
  HTML 中不仅有普通的 HTML 标签，还有 SVG 和 MathML。  
  攻击者可以构造 Payload，让过滤器在“错误的”命名空间里解析，而浏览器在实际渲染时转换到另一种命名空间，从而绕过安全检查。  

### 2.2 利用内容修改（Reparsing）问题

有时开发者在过滤后还会对 HTML 字符串进行额外替换或修改（比如使用正则删除特定模板语法），这可能会改变 HTML 结构，导致浏览器解析时和过滤器预期的不一样，从而使隐藏的恶意代码“浮出水面”。

## 3. 实际例子与代码演示

下面提供两个例子，分别展示如何利用解析器的差异和命名空间混淆进行 Mutation XSS 漏洞的构造。

### 3.1 例子 1：利用 `<title>` 标签解析差异

假设一个应用允许用户输入的内容放在 `<title>` 标签中，而过滤器没有意识到其中隐藏的 HTML 代码：

```html
<title>
  <p id="><img src onerror=alert('XSS')>">
</title>
```

**解释：**

- 过滤器可能仅当作纯文本处理，认为 `<p id="` 等只是字符串。
- 浏览器在遇到 `<title>` 标签时，会把后面的内容解析为 HTML，结果把 `<img>` 标签生成并执行 `onerror` 中的代码。

### 3.2 例子 2：利用 lxml_html_clean 库的漏洞

在某些 Python 项目中，会使用 [lxml_html_clean](https://github.com/fedora-python/lxml_html_clean) 来过滤用户输入的 HTML。下面的代码展示了如何构造一个利用 CSS 规则和注释绕过过滤器检查的 Payload（注意：仅用于安全测试研究，切勿用于非法用途）。

```python
from lxml_html_clean.lxml_html_clean import clean_html

# 构造 Payload：利用 CSS 注释隐藏危险代码
payload = """
<style>
  * {
    content: '/*';
    /* 利用 IE7 的 expression() 特性执行代码 */
    xss: expr@importession(alert("XSS"));
    content: '*/';
  }
</style>
"""

cleaned = clean_html(payload)
print("过滤后的结果：")
print(cleaned)
```

**说明：**

- 这里利用了 CSS 中 `expression()`（早期 IE 特性，会执行括号内的 JavaScript）；
- 将危险代码用 `/* ... */` 注释包裹，绕过了清理器的正则检查，但在某些浏览器（如老版本 IE）中仍能被解析执行。

### 3.3 例子 3：利用命名空间混淆

利用 SVG 或 MathML 命名空间的不同解析规则，可以将本来在安全上下文中的代码，在浏览器中转换成可执行的标签。例如：

```html
<svg>
  <style>/*<img src onerror=alert("XSS")>*/</style>
</svg>
```

**解析流程：**

- **过滤器**：看到 `<svg>` 内的 `<style>` 标签时，可能按 SVG 的规则处理，不认为其中的注释内容会生成 HTML 标签。
- **浏览器**：在渲染时把 `<style>` 标签的内容当作普通文本处理，从注释中“提取”出了 `<img>` 标签，最终导致 `onerror` 被执行。

## 4. 开发中的注意点

- **不要轻信单一的过滤器。**  
  由于 HTML 解析规则异常复杂，单一的清理器很容易因解析器差异而出现漏洞。

- **多次解析问题。**  
  如果对已经过滤过的 HTML 进行额外的替换或重新解析，有可能会破坏原有的安全结构，导致漏洞产生。

- **命名空间问题。**  
  SVG、MathML 与普通 HTML 的解析方式不同，需要格外小心处理。

> **温馨提示**  
> 上述代码和示例仅供学习和安全研究使用。实际开发中，请使用成熟且经过充分测试的安全库，并定期关注安全公告。

---

## 5. 总结

- **Mutation XSS** 利用的是过滤器与浏览器在解析 HTML 时的细微差别。  
- 利用特定标签（如 `<title>`、`<noscript>`）、命名空间（SVG、MathML）或多次解析操作，可以将看似安全的输入变为危险代码。  
- 开发安全过滤器时需要考虑各种情况，避免因解析器行为差异导致漏洞。
