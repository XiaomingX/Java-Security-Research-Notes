# 业务架构文档 (Business Architecture)

## 项目定位
本项目是一个综合性的 Java 安全漏洞研究资源库，包含多个子项目：
- Java 安全研究笔记（主项目）
- AI/ML 原型验证（ai-poc）
- 网络安全工具（amass, bugbounty）
- 漏洞 POC 集合（多个 CVE 相关子项目）

## 业务现状分析

### 核心业务领域
1. **Java 安全研究**：反序列化、Web 框架漏洞、XXE 等
2. **AI/ML 实验**：基于向量检索的文档问答系统
3. **安全工具集成**：域名枚举、漏洞扫描等
4. **漏洞复现与 POC**：各类 CVE 的验证代码

### 业务价值主张
- 为安全研究人员提供可复现的漏洞环境
- 积累安全知识库和最佳实践
- 探索 AI 在安全领域的应用可能性

## 过时业务模式识别

### [ ] 1. AI POC 项目定位模糊
**问题**：ai-poc 项目混合了多个技术栈（Python Notebooks、.NET、多个 AI 框架），缺乏清晰的业务目标
**影响**：
- 维护成本高，技术栈分散
- 难以向外部展示清晰的价值主张
- 与主项目（Java 安全）关联性弱

**建议**：
- 明确 AI POC 的业务目标：是用于安全漏洞检测？还是文档智能分析？
- 考虑将 AI 能力聚焦到"安全代码审计"或"漏洞模式识别"
- 或者独立为单独的项目，避免与 Java 安全研究混淆

### [ ] 2. 缺乏统一的知识管理体系
**问题**：各个漏洞研究分散在独立目录，缺乏横向关联和知识图谱
**影响**：
- 难以快速查找相关漏洞类型
- 无法建立漏洞之间的关联关系
- 新手学习曲线陡峭

**建议**：
- 建立统一的漏洞分类体系（按 OWASP Top 10、CWE 等标准）
- 使用 AI POC 中的向量检索能力，构建"漏洞知识问答系统"
- 添加交叉引用和相关漏洞推荐

### [ ] 3. Bug Bounty 数据集缺乏业务场景
**问题**：bugbounty/data 目录包含 91838 个 txt 文件，但缺乏使用说明和业务价值描述
**影响**：
- 数据价值不明确
- 占用大量存储空间
- 可能包含过时或无效数据

**建议**：
- 添加数据集说明文档（数据来源、更新时间、使用场景）
- 考虑数据清洗和去重
- 评估是否需要全部保留，或迁移到外部存储

### [x] 4. 项目文档已初步建立
**现状**：已有 .prompt 目录下的 agent.md 和 skill.md
**价值**：为 AI 辅助开发提供了基础框架

## 业务优化建议

### [ ] 短期优化（1-2 周）
1. 为 ai-poc 项目添加清晰的 README，说明业务目标和技术选型理由
2. 整理 bugbounty 数据集，添加使用文档
3. 在主 README 中添加"快速导航"，按漏洞类型分类

### [ ] 中期优化（1-2 月）
1. 构建漏洞知识图谱，建立漏洞之间的关联
2. 将 AI 能力应用到安全代码审计场景
3. 统一各子项目的文档风格和代码规范

### [ ] 长期规划（3-6 月）
1. 开发 Web 界面，提供漏洞检索和 POC 演示
2. 集成 CI/CD，自动化漏洞环境构建
3. 建立社区贡献机制，吸引更多安全研究者参与

## 业务指标建议

### 可衡量的业务目标
- **知识覆盖度**：覆盖 OWASP Top 10 的漏洞类型数量
- **POC 可用性**：可成功复现的 POC 比例
- **文档完整性**：有详细说明的漏洞研究比例
- **社区活跃度**：外部贡献者数量、Issue 响应时间
